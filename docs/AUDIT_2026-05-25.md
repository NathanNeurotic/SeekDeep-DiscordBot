# SeekDeep Repository Audit — 2026-05-25

Scope: public GitHub repository audit from `main`, with low-risk fixes applied on branch `audit/setup-diagnostics-20260525`.

## Problems and Fixes

### 1. First-run `.env` guidance was internally inconsistent

**Problem:** `setup_local.ps1` creates `.env` from `.env.default`, but `scripts/doctor.mjs` told users to copy `.env.example`. That gives a new user two conflicting setup paths.

**Fix applied:** `scripts/doctor.mjs` now says to run `setup_local.ps1` or copy `.env.default` to `.env`. It also explicitly checks that both `.env.default` and `.env.example` exist, with `.env.default` treated as the required first-run template and `.env.example` treated as the full reference.

### 2. `DISCORD_CLIENT_ID` was treated as required by doctor but absent from the first-run template

**Problem:** `.env.default` did not include `DISCORD_CLIENT_ID`, while `scripts/doctor.mjs` failed when it was missing. A fresh setup created by `setup_local.ps1` could therefore fail diagnostics even before the user had a chance to understand what was wrong.

**Fix applied:** `.env.default` now includes a blank `DISCORD_CLIENT_ID=` with a short explanation. `doctor.mjs` now warns, rather than hard-fails, when `DISCORD_CLIENT_ID` is absent. `DISCORD_TOKEN` remains a hard failure because the bot cannot log in without it.

### 3. `.gitignore` had duplicated and overlapping rules

**Problem:** `.gitignore` had two large publish-safety blocks with repeated entries for `.env`, `node_modules`, caches, build output, backups, logs, model folders, and generated runtime data. That makes later safety review harder because it is not obvious which rule is canonical.

**Fix applied:** `.gitignore` was consolidated into one organized file with sections for secrets, Python, Node, model caches, runtime data, agent/tool workspaces, release archives, and OS/editor noise. `.env.example` and `.env.default` are explicitly unignored.

### 4. Local/remote URL risks were not surfaced clearly in diagnostics

**Problem:** The README privacy note is careful about local-vs-remote services, but `doctor.mjs` did not warn when `LOCAL_AI_BASE_URL` or `SEARXNG_BASE_URL` pointed away from loopback.

**Fix applied:** `doctor.mjs` now warns if either endpoint is not `127.0.0.1` or `localhost`, because prompts/media/search queries may leave the local machine in that configuration.

### 5. Doctor output treated warnings and failures too similarly

**Problem:** The old doctor script only distinguished hard failures at exit, but its success message said setup was ready even when warnings existed.

**Fix applied:** `doctor.mjs` now has a distinct completion message for warnings. It still exits zero when only warnings are present, because offline local services may legitimately be down during setup checks.

## Suggestions / Improvements

### High priority

1. **Run full local preflight before merging.** Use:

   ```powershell
   cd C:\Users\natha\SeekDeep-DiscordBot
   git fetch origin
   git checkout audit/setup-diagnostics-20260525
   npm install
   npm run preflight
   npm run doctor
   ```

   I could not run the repo locally from this environment because GitHub/network cloning is unavailable in the execution container. The GitHub connector allowed safe file edits and review, but not actual Node/Python execution.

2. **Add `package-lock.json`.** This is a public installable app with Node dependencies. Without a lockfile, `npm install` can resolve newer transitive versions over time. A checked-in lockfile makes fresh installs and bug reports reproducible.

3. **Add CI for `npm run preflight`.** The repo already has a strong `scripts/preflight.mjs`; running it on pull requests would prevent syntax regressions in `index.js`, `gui/*.js`, `local_ai_server.py`, `warmup_local_cache.py`, and `gui_endpoints.py` from landing unnoticed.

4. **Split `index.js` by subsystem.** The README describes `index.js` as roughly 16k lines. It currently holds bot startup, logging, GUI event forwarding, memory, archive, routing, image workflow, context menus, and Discord event handlers in one file. Split into modules such as `src/config`, `src/logging`, `src/discord`, `src/routing`, `src/archive`, `src/image`, `src/memory`, and `src/web`. This is the biggest maintainability improvement.

5. **Make feature-flag docs match shipped defaults.** `.env.default` and `.env.example` disagree on some defaults, especially optional image-editing features. That may be intentional, but it should be documented as “first-run safe defaults” versus “full capability reference defaults,” otherwise new users may not know which file represents recommended behavior.

### Medium priority

1. **Introduce schema-based env validation.** Both Node and Python parse many env vars independently. A small env schema would prevent drift between `.env.default`, `.env.example`, README, doctor, and runtime behavior.

2. **Add a generated env reference check.** A script can compare env keys found in `index.js`, `local_ai_server.py`, `gui_endpoints.py`, and docs against `.env.example`/`.env.default`. This would catch missing or stale config keys automatically.

3. **Reduce inline smoke-test mirrors.** `smoke_test.mjs` intentionally imports real helpers for many checks, but still contains some inline regex mirrors. Any mirrored logic can drift. Prefer exporting test-only helpers behind `SEEKDEEP_TEST_MODE=1` when practical.

4. **Standardize boolean parsing.** Node and Python both parse flags like `on/true/1/yes`, but not always with identical helper functions. Centralize this per language and use it everywhere.

5. **Add a startup redaction test.** Logging redaction exists in `index.js`, but it should be smoke-tested against Discord token, `hf_`, `sk-`, `nvapi-`, bearer header, email, and Windows path examples.

## Dead Code to Remove Safely

These are removal candidates, not changes applied in this branch. Verify with local grep and `npm run preflight` before deleting.

1. **Removed archive command compatibility path.** `seekdeepHandleRemovedArchiveCommandMessage()` appears to exist only to tell users that a removed archive command was removed. If the old command is no longer being invoked by real users, remove the handler and its pre-address route.

2. **Real-ESRGAN scaffold until actually supported.** Real-ESRGAN is feature-flagged off and dependency-gated. If it is not planned soon, keep the `lanczos` path and remove or isolate the Real-ESRGAN selection/check code into a separate optional module.

3. **Duplicate GUI/static documentation copies.** The repo appears to contain both `gui/*.html` and `docs/*.html` pages. Confirm whether `docs/` is the GitHub Pages mirror. If it is manually copied from `gui/`, add a sync script or document the source of truth. If one side is stale, remove the stale copy.

4. **Old handoff and planning docs.** Files like `HANDOFF_CLAUDE_CODE.md`, `PLANNED.md`, and old release notes may still be useful, but if they describe already-completed work they should move under `docs/archive/` or be trimmed so the repo front page stays focused.

5. **Status-bank joke entries with privacy implications.** Some rotating statuses joke about watching/search history/webcam. Even as jokes, they conflict with the README’s privacy posture. Replace them with harmless local-model/VRAM/task-queue jokes before broad public promotion.
