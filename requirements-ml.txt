# SeekDeep · ML deps — heavy libraries for /chat (local), /image, /vision,
# and the warmup_local_cache.py model puller.
#
# Installed on first use via the in-app "Install ML libraries" prompt
# (POST /deps/install endpoint). NOT installed by the Tauri sidecar's
# first-run flow — the boot deps in requirements-local.txt are enough to
# bring the server up; ML deps are lazy because their install is ~2 GB
# and ~5-10 minutes, which is a bad first-launch experience for users
# who only want to play with the GUI / remote backends.
#
# Remote-only setups (openai-compat / anthropic / gemini) never need this
# file — the /chat handler routes to HTTP for those backends without ever
# importing torch / transformers / diffusers.

# PyTorch's CUDA wheels live behind a separate index. We previously
# used --extra-index-url which is permissive (pip can resolve from
# EITHER PyPI or the extra URL) and the resolver kept preferring the
# plain PyPI torch over torch+cu128 — users ended up with a CPU-only
# wheel even though they explicitly asked for ML deps. Pin the local
# version (+cu128) so pip MUST pull from pytorch.org. The trio is
# version-locked together by PyTorch so we pin the same triple.
#
# When PyTorch ships a newer toolkit (cu130+), bump these three lines
# and the cu128 URL. The wizard's "wrong wheel" FIX path uses
# pip_install_torch_variant() which handles cu118/cu121/cu124/cu126/cu128
# at runtime — this file just sets the default for fresh installs.
--extra-index-url https://download.pytorch.org/whl/cu128
# Verified-available on https://download.pytorch.org/whl/cu128/ as of 2026-05:
#   torch       2.11.0+cu128  (latest cu128 — cp310..cp314)
#   torchvision 0.26.0+cu128  (matches torch 2.11)
# torchaudio is INTENTIONALLY OMITTED — SeekDeep imports torch +
# torchvision (via diffusers/transformers) but never torchaudio. Pinning
# it caused a ResolutionImpossible: torchaudio 2.10.0+cu128 hard-pins
# torch==2.10.0, which conflicts with our torch 2.11.0 pin. Keep it out
# until SeekDeep actually grows an audio surface.
torch==2.11.0+cu128 ; platform_system == "Windows"
torchvision==0.26.0+cu128 ; platform_system == "Windows"
# AUD-007: non-Windows (Linux/macOS) was previously UNPINNED — `pip install -r`
# would pull whatever torch/torchvision the index currently serves, so a future
# breaking release could silently land and the first failure would be a user's
# machine, not CI (CI doesn't install full ML deps). Bound both so the platform
# wheel resolver (CPU/MPS on macOS, CPU/CUDA on Linux) still picks an appropriate
# build, while the upper cap stops a breaking major/minor from auto-installing.
# torchvision hard-pins its torch, so bounding the pair keeps them consistent
# (e.g. torch 2.11 <-> torchvision 0.26, torch 2.4 <-> torchvision 0.19).
# Unsupported CUDA/CPU platform? Override explicitly, e.g.:
#   pip install "torch>=2.6,<2.12" torchvision --index-url https://download.pytorch.org/whl/cpu
#   pip install "torch>=2.6,<2.12" torchvision --index-url https://download.pytorch.org/whl/cu124
torch>=2.4,<2.12 ; platform_system != "Windows"
torchvision>=0.19,<0.27 ; platform_system != "Windows"
# FIX-3: upper bounds cap the major so a future breaking release can't
# silently install on `pip install -r`. Bounds chosen to NOT exclude the
# known-good versions running in the dev venv as of 2026-05 (transformers
# 5.8.1, diffusers 0.39.0.dev0, accelerate 1.13.0, bitsandbytes 0.49.2) —
# the audit's suggested transformers<5 / diffusers<0.40 would have
# force-downgraded the working install, so cap at the next major instead.
accelerate>=1.2.0,<2
bitsandbytes>=0.45.0,<1.0
transformers>=4.51.0,<6
diffusers>=0.33.0,<1.0
safetensors>=0.5.0,<1.0
# Both sentencepiece + tiktoken: transformers needs ONE of these to
# instantiate slow tokenizers when the fast (rust `tokenizers`) backend
# can't parse a model's tokenizer.json. Different model families use
# different conventions:
#   - sentencepiece: Llama 2, Mistral, Gemma, IBM Granite (older)
#   - tiktoken:      Llama 3.x, Qwen 2.5+, IBM Granite 3.x
# We require both as belt-and-suspenders so AutoTokenizer.from_pretrained
# never bails with "You need to have sentencepiece or tiktoken installed
# to convert a slow tokenizer to a fast one." — the exact ValueError
# users hit with Llama 3.1 / Granite 3.3 (tokenizer-load-failure 503).
sentencepiece>=0.2.0
tiktoken>=0.8.0
protobuf>=7.35.0
einops>=0.8.0
timm>=1.0.15
open-clip-torch>=3.0.0
qwen-vl-utils>=0.0.8
imageio>=2.36.0
imageio-ffmpeg>=0.5.1
